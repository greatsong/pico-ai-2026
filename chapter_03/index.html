    전자담배 검출 시스템 설계 – 센서→판단→출력 통합 프로젝트¶

    3차시(50분) | 임계값(Threshold) | 조건문(if/elif/else) | 난이도 ★★★☆☆

    학습목표: 1. MQ2 센서와 RGB LED Stick을 하나의 시스템으로 통합하여 검출 시스템을 완성할 수 있다 2. 임계값(threshold) 개념을 이해하고, 조건문으로 상태 판단 로직을 작성할 수 있다

    오늘의 질문: "센서가 읽은 숫자를, 사람이 직관적으로 이해할 수 있는 '빛의 신호'로 바꾸려면 어떤 논리가 필요할까?"

    🗺️ 전체 흐름 속 오늘의 위치¶

    여러분은 지금 3차시 프로젝트의 대미(大尾)에 도달했습니다. 1차시에서 배운 입력(MQ2 가스 센서)과 2차시에서 배운 출력(RGB LED Stick)을 하나의 시스템으로 결합하는 시간입니다.

    flowchart LR A["🔬 1차시<br/>센서로 세상 읽기<br/>(입력)"] --> B["💡 2차시<br/>빛으로 말하기<br/>(출력)"] B --> C["🚨 3차시<br/>검출 시스템 설계<br/>(통합)"] style A fill:#e3f2fd,stroke:#1976D2,color:#000 style B fill:#e8f5e9,stroke:#388E3C,color:#000 style C fill:#fff3e0,stroke:#F57C00,stroke-width:3px,color:#000

    1차시의 gas_sensor.read_u16() + 2차시의 np[i] = (R, G, B) → 오늘 하나로 합친다!

    🎯 핵심 개념 1: 임계값(Threshold)¶

    "몇 도부터 열이 나는 걸까?" — 체온계에서 37.5°C가 기준이듯, 센서 값에도 '여기부터 위험'이라는 경계선이 필요합니다. 이 경계선을 임계값이라고 합니다.

    🔧 핵심 개념 2: 센서→판단→출력¶

    센서가 숫자를 읽고(입력), 조건문이 상태를 판단하고(처리), LED가 색으로 알려준다(출력). 이 3단계 구조가 모든 IoT 시스템의 뼈대입니다.

    💬 사회적 맥락: 학교 전자담배 문제¶

    전자담배는 연기가 적고 냄새가 금방 사라져 적발이 어렵습니다. 하지만 가스 성분은 남아있습니다. 미국 학교에 설치된 'HALO 센서'(수백만 원)와 같은 원리를 만 원 이하 부품으로 구현해 봅시다.

    🚀 진로 연결¶

    오늘 만드는 시스템은 IoT 엔지니어, 스마트시티 설계자, 공중보건 데이터 분석가가 실제로 다루는 기술의 축소판입니다.

    수업 흐름 (50분)¶

    1단계: 도입 — 왜 만드는가? (5분)¶

    사회적 맥락 연결 + 임계값 개념 도입

    2단계: 실습 1 — 회로 연결 + 검출 시스템 v1 (20분)¶

    MQ2 + LED Stick 동시 연결, 조건문으로 초록/노랑/빨강 전환

    3단계: 실습 2 — 검출 시스템 v2 + 경보 애니메이션 (15분)¶

    깜빡임·숨쉬기 효과 추가, 사용자 심리를 고려한 시각 피드백

    4단계: 정리 — 캘리브레이션 + 성찰 + 확장 토의 (10분)¶

    임계값 튜닝, 시스템 구조 다이어그램, 바이브 코딩 챌린지, 형성 평가

    1단계: 도입 — 왜 만드는가? (00:00~05:00)¶

    학교 화장실, 보이지 않는 위험¶

    전자담배의 가장 큰 문제는 적발의 어려움입니다.

    구분 일반 담배 전자담배
    연기 많고 오래 남음 적고 빨리 사라짐
    냄새 강하고 지속 약하고 금방 소멸
    가스 성분 남아있음 남아있음 ✅

    핵심 포인트: 냄새는 사라져도, 가스 성분(글리세린, 프로필렌글리콜)은 공기 중에 남아있습니다. MQ2 센서가 바로 이 성분을 감지할 수 있습니다.

    [그림 1] 전자담배의 연기는 사라져도 가스 성분은 남아 있다 — MQ2 센서가 이를 감지한다

    실제 상용 제품: HALO 센서¶

    미국 학교에 설치된 HALO Smart Sensor는 공기질, 소음, 전자담배 연기를 동시에 감지합니다. 가격은 수백만 원. 오늘 우리는 같은 원리를 만 원 이하 부품으로 구현합니다.

    핵심 개념: 임계값(Threshold)¶

    비유: 체온계의 37.5°C를 생각해 보세요. 체온이 36.5°C면 정상, 37.5°C 이상이면 "열이 있다"고 판단하죠? 37.5°C가 바로 임계값입니다.

    우리 검출 시스템에도 이런 기준선이 두 개 필요합니다.

    flowchart LR subgraph 센서값 범위 direction LR A["0"] --- B["THRESHOLD_WARN<br/>(주의 기준)"] --- C["THRESHOLD_ALERT<br/>(경보 기준)"] --- D["65535"] end A -.- E["🟢 안전<br/>초록 LED"] B -.- F["🟡 주의<br/>노랑 LED"] C -.- G["🔴 경보<br/>빨강 LED"] style E fill:#4caf50,color:#fff style F fill:#ffeb3b,color:#000 style G fill:#f44336,color:#fff
    상태 센서값 범위 LED 색상 의미
    🟢 안전 0 ~ THRESHOLD_WARN 미만 초록 평상시, 이상 없음
    🟡 주의 THRESHOLD_WARN ~ THRESHOLD_ALERT 미만 노랑 뭔가 감지됨, 주시 필요
    🔴 경보 THRESHOLD_ALERT 이상 빨강 확실히 감지됨, 조치 필요

    ⚠️ 중요: 임계값은 환경마다 다릅니다! 교실 온도, 습도, 환기 상태에 따라 평상시 센서값이 다르기 때문입니다. "정답"은 없고, 각자 자기 환경에 맞게 조정해야 합니다.

    자기점검 퀴즈 1

    Q1. 임계값(threshold)이란 무엇인가요?

    ① 센서가 측정할 수 있는 최대값 ② 상태를 구분하는 기준이 되는 경계값 ③ LED의 최대 밝기를 결정하는 값 ④ ADC가 변환할 수 있는 비트 수

    정답 확인

    정답: ②

    해설: 임계값은 "여기부터 다른 상태로 전환한다"는 경계선입니다. 체온계에서 37.5°C가 '정상'과 '발열'을 나누는 기준이듯, 우리 시스템에서는 THRESHOLD_WARN과 THRESHOLD_ALERT가 '안전', '주의', '경보'를 나누는 기준입니다. 잘 맞혔다면 훌륭합니다! 👏

    Q2. 임계값이 환경마다 다른 이유로 가장 적절하지 않은 것은?

    ① 교실 온도가 다르면 센서 기본값이 달라진다 ② 환기 상태에 따라 공기 중 가스 농도가 다르다 ③ MQ2 센서는 디지털 센서라서 오차가 생긴다 ④ 센서 개체별로 미세한 성능 차이가 있다

    정답 확인

    정답: ③

    해설: MQ2는 아날로그 센서입니다(1차시 복습!). 디지털 센서가 아니라 아날로그 신호를 ADC로 변환하는 구조이죠. 나머지 ①②④는 모두 임계값이 환경마다 달라지는 실제 이유입니다. 틀렸다면 1차시의 아날로그/디지털 개념을 복습해 보세요.

    2단계: 실습 1 — 회로 연결 + 검출 시스템 v1 (05:00~25:00)¶

    회로 연결 (05:00~08:00)¶

    MQ2 센서와 RGB LED Stick, 두 부품을 동시에 연결합니다. 1차시의 센서 + 2차시의 LED를 합치는 순간입니다!

    Grove Shield 포트 Pico 핀 연결 부품 포트 유형 역할
    A0 GP26 MQ2 가스 센서 아날로그(ADC) 가스 농도 읽기
    D16 GP16 RGB LED Stick 디지털(NeoPixel) 색상 출력
    flowchart TB subgraph SHIELD["Grove Shield (Pico 2 WH 장착)"] A0["A0 포트<br/>아날로그 입력"] D16["D16 포트<br/>디지털 출력"] end MQ2["🔬 MQ2 가스 센서<br/>공기 중 가스 감지"] LED["💡 RGB LED Stick<br/>10개 NeoPixel LED"] A0 -->|"GP26 (ADC)"| MQ2 D16 -->|"GP16 (Digital)"| LED style SHIELD fill:#e8eaf6,stroke:#3f51b5,color:#000 style MQ2 fill:#fff3e0,stroke:#ff9800,color:#000 style LED fill:#e8f5e9,stroke:#4caf50,color:#000

    💡 꿀팁: MQ2를 먼저 A0에 꽂고 USB를 연결하세요. 센서가 예열되는 동안(1~2분) LED Stick을 D16에 연결하면 시간을 아낄 수 있습니다!

    ⚠️ 안전 주의: LED Stick의 IN 쪽이 Shield에 연결되어야 합니다. IN/OUT 표시가 작으니 잘 확인하세요. 반대로 꽂으면 LED가 켜지지 않습니다.

    센서→판단→출력 흐름 이해하기¶

    코드를 작성하기 전에, 시스템의 논리 흐름을 먼저 이해합시다.

    flowchart TD A["센서값 읽기<br/>read_average()"] --> B{"value >= THRESHOLD_ALERT?"} B -->|"Yes"| C["🔴 경보<br/>빨강 LED"] B -->|"No"| D{"value >= THRESHOLD_WARN?"} D -->|"Yes"| E["🟡 주의<br/>노랑 LED"] D -->|"No"| F["🟢 안전<br/>초록 LED"] C --> G["0.5초 대기"] E --> G F --> G G --> A style B fill:#ffcdd2,stroke:#c62828,color:#000 style D fill:#fff9c4,stroke:#f9a825,color:#000 style C fill:#f44336,color:#fff style E fill:#ffeb3b,color:#000 style F fill:#4caf50,color:#fff

    이 흐름도에서 마름모(◇)가 바로 조건문(if/elif/else)입니다. 위에서부터 순서대로 검사하며, 가장 먼저 True가 되는 조건에서 멈춥니다.

    🔑 왜 ALERT를 먼저 검사할까? if문은 위에서부터 순서대로 확인합니다. 만약 WARN을 먼저 검사하면, 경보 수준(예: 40000)도 WARN 조건(≥ 20000)에 걸려서 노란색으로 표시됩니다! 가장 심각한 조건을 먼저 검사하는 것이 원칙입니다.

    검출 시스템 v1: 기본 색상 전환¶

    이제 코드를 작성합니다. 1차시의 센서 읽기 코드와 2차시의 LED 제어 코드를 합칩니다.

    # === 전자담배 검출 시스템 v1 ===
# MQ2 센서 값에 따라 LED 색상이 바뀌는 검출기
# 파일명: step1_detector_v1.py

from machine import ADC, Pin
from neopixel import NeoPixel
import time

# --- 하드웨어 설정 ---
gas_sensor = ADC(Pin(26))        # A0: MQ2 가스 센서  <- 1차시에서 배운 부분
NUM_LEDS = 10
np = NeoPixel(Pin(16), NUM_LEDS) # D16: RGB LED Stick <- 2차시에서 배운 부분

# --- 임계값 설정 ---
# ※ 환경에 따라 조정 필요! 아래는 시작값입니다
THRESHOLD_WARN = 20000    # 이 이상이면 주의 (노랑)  <- 여기가 임계값!
THRESHOLD_ALERT = 35000   # 이 이상이면 경보 (빨강)  <- 여기가 임계값!

# --- 이동 평균 함수 (1차시 복습) ---
def read_average(sensor, count=10):
    """10번 읽어서 평균 → 노이즈 감소"""
    total = 0
    for i in range(count):
        total += sensor.read_u16()
        time.sleep_ms(10)
    return total // count

# --- LED 전체를 한 색으로 설정하는 함수 ---
def set_all_leds(r, g, b):
    """LED 10개를 모두 같은 색으로 설정"""
    for i in range(NUM_LEDS):
        np[i] = (r, g, b)
    np.write()  # 반드시 write()!

# --- 메인 루프 ---
print("=== 전자담배 검출 시스템 v1 가동 ===")
print(f"주의 임계값: {THRESHOLD_WARN}")
print(f"경보 임계값: {THRESHOLD_ALERT}")

while True:
    value = read_average(gas_sensor)

    # 조건문으로 상태 판단 (가장 심각한 것부터!)
    if value >= THRESHOLD_ALERT:       # <- 경보 조건을 먼저!
        set_all_leds(255, 0, 0)        # 🔴 빨강
        status = "🔴 경보"
    elif value >= THRESHOLD_WARN:      # <- 그 다음 주의
        set_all_leds(255, 255, 0)      # 🟡 노랑
        status = "🟡 주의"
    else:                              # <- 나머지는 안전
        set_all_leds(0, 255, 0)        # 🟢 초록
        status = "🟢 안전"

    print(f"{status} | 센서값: {value}")
    time.sleep(0.5)

    예상 출력: 

    === 전자담배 검출 시스템 v1 가동 ===
주의 임계값: 20000
경보 임계값: 35000
🟢 안전 | 센서값: 12450
🟢 안전 | 센서값: 12380
🟡 주의 | 센서값: 24500
🔴 경보 | 센서값: 38200
🟡 주의 | 센서값: 28100
🟢 안전 | 센서값: 13200

    코드 해부: 무엇이 새롭고, 무엇이 복습인가?¶

    줄 번호 코드 차시 역할
    7행 gas_sensor = ADC(Pin(26)) 1차시 복습 MQ2 센서 준비
    9행 np = NeoPixel(Pin(16), NUM_LEDS) 2차시 복습 LED Stick 준비
    12~13행 THRESHOLD_WARN = 20000 오늘 새로 배움 임계값 설정
    28~35행 if/elif/else 블록 오늘 새로 배움 상태 판단 로직
    22~25행 set_all_leds() 2차시 복습 LED 색상 변경

    1차시 + 2차시에서 배운 코드가 그대로 활용됩니다. 새로 추가된 건 임계값 2줄과 if/elif/else 조건문 뿐입니다!

    🐛 에러 경험 설계: 이 코드에서 뭐가 잘못됐을까?¶

    다음 코드를 보고, 실행하면 어떤 문제가 생길지 예측해 보세요.

    # ❌ 잘못된 코드 — 어떤 문제가 있을까요?
while True:
    value = read_average(gas_sensor)

    if value >= THRESHOLD_WARN:        # 주의를 먼저 검사!
        set_all_leds(255, 255, 0)      # 노랑
        status = "🟡 주의"
    elif value >= THRESHOLD_ALERT:     # 경보를 나중에 검사
        set_all_leds(255, 0, 0)        # 빨강
        status = "🔴 경보"
    else:
        set_all_leds(0, 255, 0)
        status = "🟢 안전"
    문제점과 해설 보기 **문제**: 센서값이 40000(경보 수준)이어도 **노란색**만 나옵니다! **원인**: `if value >= THRESHOLD_WARN` (20000)을 먼저 검사하므로, 40000은 당연히 20000 이상이라 여기서 걸립니다. `elif`는 이미 위에서 True가 나왔으므로 검사하지도 않습니다. **교훈**: 조건문은 **가장 엄격한(높은) 조건부터 먼저** 검사해야 합니다! **수정**: `THRESHOLD_ALERT`를 먼저 검사하도록 if와 elif의 순서를 바꿔야 합니다(v1 코드처럼).

    실습 중 자주 묻는 질문¶

    증상 원인 해결
    LED가 처음부터 노랑/빨강 평상시 값이 이미 임계값 초과 Thonny 셸의 센서값 확인 → THRESHOLD 값 올리기
    입김 불어도 색 안 변함 임계값이 너무 높음 THRESHOLD_WARN을 '평상시 값 + 3000'으로 낮추기
    LED 안 켜짐 D16 포트 미연결 / np.write() 누락 케이블 확인 + 코드에 np.write() 있는지 확인
    ImportError: no module named 'neopixel' 펌웨어 버전 문제 MicroPython 최신 펌웨어로 업데이트

    자기점검 퀴즈 2

    Q1. 다음 중 검출 시스템 v1의 if/elif/else 구조에 대한 설명으로 올바른 것은?

    ① if, elif, else 조건은 동시에 모두 실행된다 ② elif는 위의 if가 False일 때만 검사된다 ③ else는 if가 True일 때 실행된다 ④ 조건의 순서는 결과에 영향을 주지 않는다

    정답 확인

    정답: ②

    해설: if/elif/else는 위에서부터 순서대로 검사합니다. if가 True이면 elif와 else는 건너뛰고, if가 False일 때만 elif를 검사합니다. ④가 틀린 이유는 바로 위의 에러 경험 설계에서 확인했죠? 순서가 결과를 완전히 바꿔버립니다! 맞혔다면 조건문의 핵심을 정확히 이해한 것입니다 👍

    Q2. (빈칸 채우기) 센서값이 42000이고, THRESHOLD_WARN=20000, THRESHOLD_ALERT=35000일 때, LED 색상은 ______이다.

    정답 확인

    정답: 빨강(경보)

    해설: 42000 ≥ 35000(THRESHOLD_ALERT)이므로 첫 번째 if 조건이 True가 되어 set_all_leds(255, 0, 0) 즉 빨간색이 켜집니다. elif와 else는 실행되지 않습니다.

    검출 시스템 v2: 경보 애니메이션 추가¶

    v1 코드에서 바뀐 부분에 집중하면서 읽어보세요.

    # === 전자담배 검출 시스템 v2 (경보 애니메이션) ===
# v1에서 추가: blink_red(), breathe_yellow(), 안전 밝기 조절
# 파일명: step2_detector_v2.py

from machine import ADC, Pin
from neopixel import NeoPixel
import time

# --- 하드웨어 설정 (v1과 동일) ---
gas_sensor = ADC(Pin(26))
NUM_LEDS = 10
np = NeoPixel(Pin(16), NUM_LEDS)

# --- 임계값 설정 (v1과 동일) ---
THRESHOLD_WARN = 20000
THRESHOLD_ALERT = 35000

# --- v1에서 가져온 함수 (동일) ---
def read_average(sensor, count=10):
    total = 0
    for i in range(count):
        total += sensor.read_u16()
        time.sleep_ms(10)
    return total // count

def set_all_leds(r, g, b):
    for i in range(NUM_LEDS):
        np[i] = (r, g, b)
    np.write()

# ========== v2에서 새로 추가된 함수 ==========

def blink_red():
    """🔴 빨강 깜빡임 — 위험 상황에서 주의를 끄는 효과
    켜고 0.15초, 끄고 0.15초 = 약 0.3초에 한 번 깜빡임"""
    set_all_leds(255, 0, 0)   # 켜기
    time.sleep(0.15)
    set_all_leds(0, 0, 0)     # 끄기
    time.sleep(0.15)

def breathe_yellow(step):
    """🟡 노랑 숨쉬기 — 밝기가 서서히 변하는 효과
    step: 0~19 사이 값, 호출할 때마다 1씩 증가시켜 사용

    원리: 삼각파 (0→250→0 반복)
    step 0~9:  점점 밝아짐 (0, 25, 50, ... 225, 250)
    step 10~19: 점점 어두워짐 (250, 225, ... 25, 0)
    """
    if step < 10:
        brightness = step * 25          # 0→250
    else:
        brightness = (19 - step) * 25   # 250→0
    set_all_leds(brightness, brightness, 0)  # R=G → 노란색 계열

# ========== 메인 루프 (v1에서 업그레이드) ==========
print("=== 전자담배 검출 시스템 v2 ===")

breath_step = 0  # 숨쉬기 애니메이션의 현재 단계

while True:
    value = read_average(gas_sensor)

    if value >= THRESHOLD_ALERT:
        blink_red()                    # <- v1: set_all_leds(255,0,0) → v2: 깜빡임!
        status = "🔴 경보"

    elif value >= THRESHOLD_WARN:
        breathe_yellow(breath_step)    # <- v1: set_all_leds(255,255,0) → v2: 숨쉬기!
        breath_step = (breath_step + 1) % 20  # 0~19 순환 카운터
        status = "🟡 주의"
        time.sleep(0.1)

    else:
        set_all_leds(0, 50, 0)         # <- v1: (0,255,0) → v2: (0,50,0) 은은하게!
        status = "🟢 안전"
        breath_step = 0                # 숨쉬기 단계 초기화
        time.sleep(0.5)

    print(f"{status} | 값: {value}")

🟢 안전 | 값: 12450
🟡 주의 | 값: 23800
🟡 주의 | 값: 25100

    v1 → v2 변경 사항 정리¶

    변경 위치 v1 코드 v2 코드 이유
    경보 시 set_all_leds(255, 0, 0) blink_red() 깜빡임으로 긴급함 전달
    주의 시 set_all_leds(255, 255, 0) breathe_yellow(step) 숨쉬기로 불안감 표현
    안전 시 (0, 255, 0) — 눈부심 (0, 50, 0) — 은은함 24시간 켜둬도 눈 편함
    새 변수 없음 breath_step 숨쉬기 단계 추적용

    핵심 코드 해설: % 20의 비밀¶

    breath_step = (breath_step + 1) % 20  # 0~19 순환 카운터

    이 한 줄이 숨쉬기 효과의 핵심입니다.

    breath_step 값 변화 밝기 (step × 25)
    0 → 1 → 2 → ... → 9 점점 증가 0 → 25 → 50 → ... → 225
    10 → 11 → ... → 19 점점 감소 250 → 225 → ... → 0
    19 → 0 (% 20으로 순환) 다시 시작 0 (처음으로)

    % 20은 나머지 연산입니다. 19 다음에 (19+1) % 20 = 0이 되므로, 0~19 사이를 무한히 순환합니다. 마치 시계가 12 다음에 다시 1로 돌아오는 것과 같습니다.

    Q1. v2에서 안전 상태의 LED 색상을 (0, 255, 0) 대신 (0, 50, 0)으로 바꾼 이유는?

    ① 초록색이 아닌 다른 색을 표현하려고 ② 전력 소모를 줄이려고 ③ 평상시에는 은은하게, 경보 때만 강렬하게 하려고 ④ MicroPython에서 255를 사용하면 오류가 나서

    정답: ③

    해설: 24시간 밝은 빛은 눈에 피로하고, 경보와의 대비도 약해집니다. 실제 제품에서도 "평상시 은은, 위험 시 강렬" 원칙을 따릅니다. ②도 부분적으로 맞지만 주된 이유는 사용자 경험(UX) 설계입니다.

    Q2. (참/거짓) blink_red() 함수 안에 이미 time.sleep(0.15)가 포함되어 있으므로, 메인 루프의 경보 분기에서 별도의 time.sleep()이 필요 없다.

    정답 확인

    정답: 참 ✅

    해설: blink_red() 안에 sleep(0.15)가 두 번(켜기+끄기) 들어있어 함수 한 번 호출에 약 0.3초가 소요됩니다. 별도의 sleep 없이도 적절한 속도로 깜빡입니다. 맞혔다면 함수의 내부 동작을 잘 추적하고 있는 것입니다! 🎉

    형성 평가¶

    [문항 1] 시스템 구조 이해 (기본)

    전자담배 검출 시스템의 3단계 구조를 올바르게 나열한 것은?

    ① 출력 → 처리 → 입력 ② 입력 → 출력 → 처리 ③ 입력 → 처리 → 출력 ④ 처리 → 입력 → 출력

    정답 확인 **정답: ③** **해설**: MQ2 센서가 가스 농도를 **입력**받고, if/elif/else 조건문이 상태를 **처리**(판단)하고, RGB LED Stick이 색상으로 **출력**합니다. 이 순서는 모든 IoT 시스템의 기본 구조입니다.

    [문항 2] 임계값 설정 근거 (응용)

    학생 A의 교실에서 MQ2 센서의 평상시 값이 약 15000으로 측정되었습니다. THRESHOLD_WARN과 THRESHOLD_ALERT를 어떻게 설정하는 것이 적절한지 근거와 함께 서술하세요.

    모범 답안 확인 **모범 답안**: - THRESHOLD_WARN = **20000** (기준값 15000 + 5000) - THRESHOLD_ALERT = **30000** (기준값 15000 + 15000) **근거**: 평상시 값(15000)에서 센서 노이즈(±2000 정도)를 고려하면, 주의 임계값은 노이즈 범위 밖인 +5000 정도가 적절합니다. 경보 임계값은 확실한 감지를 의미하므로 +15000 정도로 여유를 둡니다. 너무 낮으면 오탐(false positive)이 많아지고, 너무 높으면 실제 감지를 놓칠 수 있으므로 균형이 중요합니다. **채점 기준**: 기준값 기반 설정(2점) + 오탐/미탐 균형 언급(2점) + 구체적 수치 제시(1점)

    [문항 3] 비판적 사고 + 설계 역량 (심화)

    "이 전자담배 검출 시스템의 한계점 2가지를 지적하고, 각각에 대한 구체적인 개선 방안을 제안하시오."

    모범 답안 확인 **한계점 1**: MQ2 센서가 전자담배뿐 아니라 손소독제, 헤어스프레이 등 다른 화학물질에도 반응하여 **오탐(false positive)**이 발생할 수 있다. → **개선**: CO2 전용 센서(SCD30)를 추가하여 다중 센서 퓨전으로 정확도를 높인다. 두 센서가 동시에 반응할 때만 경보를 울리도록 AND 조건 로직을 구현한다. **한계점 2**: 고정 임계값은 시간대(점심 후 환기 부족)나 계절(겨울 창문 닫힘)에 따라 **오작동**할 수 있다. → **개선**: 자동 캘리브레이션 기능을 추가한다. 매 5분마다 최근 평균값을 갱신하여 임계값을 동적으로 조정하는 적응형 알고리즘을 적용한다. **채점 기준**: 한계점 2가지(각 2점) + 구체적 개선 방안(각 3점) = 총 10점

    [문항 4] 코드 읽기 (기본)

    다음 코드에서 breath_step = (breath_step + 1) % 20의 역할을 설명하세요.

    정답 확인 **정답**: breath_step 변수를 1씩 증가시키되, 20이 되면 자동으로 0으로 돌아오게 하는 **순환 카운터**입니다. % 연산자(나머지 연산)를 사용하여 0~19 범위를 무한히 반복하며, 이를 통해 숨쉬기 애니메이션의 밝기가 점점 밝아졌다(step 0~9) 점점 어두워지는(step 10~19) 효과를 만듭니다.

    [문항 5] 설계 사고 (응용)

    여러분이 이 검출 시스템을 실제 학교 화장실에 설치한다고 가정할 때, 기술적 측면 외에 고려해야 할 윤리적·사회적 문제를 1가지 이상 제시하세요.

    예시 답안 확인 **예시 답안들**: - **프라이버시**: 화장실은 사적 공간이므로, 센서 데이터가 개인 식별에 사용되지 않도록 해야 한다. 누가 사용 중이었는지가 아닌, 공기질 데이터만 기록해야 한다. - **오탐 시 피해**: 손소독제 사용 등으로 오탐이 발생했을 때, 무고한 학생이 의심받을 수 있다. 경보 데이터만으로 처벌하면 안 되고 추가 확인이 필요하다. - **감시 사회 우려**: 기술적 해결만 추구하면 학교가 감시 공간이 될 수 있다. 교육과 상담이 기술보다 우선되어야 한다. **핵심**: 기술은 도구일 뿐, 어떻게 사용하느냐가 더 중요합니다!
